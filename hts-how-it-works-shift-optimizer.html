The Technology

2 layers working together

Provably optimal — not trial and error

Constraint Programming — OR-Tools CP-SAT

Google's CP-SAT solver assigns 651 binary variables simultaneously using constraint propagation and branch-and-bound search. The result is a mathematically proven optimal schedule — not a heuristic approximation. The same solver powers Google's workforce scheduling and vehicle routing.

Hard rules guaranteed + soft preferences maximized

2-Stage Objective — Penalty + Wage

Minimize: Penalty × 10,000 + Weekly wage. A single constraint violation (10,000 pts) outweighs the maximum possible wage difference (~$200) by 50×. Compliance always comes first. Within equal penalty levels, the cheapest valid schedule wins.

Constraint Design

Hard rules vs. soft preferences

Which rules must never be broken, and which are best-effort goals?

Phase 1 — Core Constraints

HARD

Must-off days

Absolute unavailability — solver never assigns a shift on these days.

HARD

Demand staffing minimums

Each shift must meet minimum headcount per day type (Normal/Sale/Holiday). Returns infeasible if impossible.

HARD

Leader per shift

Every shift needs ≥1 leader (Manager or SL). At least 1 Manager must appear somewhere each day.

HARD

Skill coverage

Each shift: Cashier Basic ×N (default 2), Food Advanced ×1, Deli Advanced ×1, Food Basic ×1.

HARD

Max consecutive days

No employee works more than N consecutive days (default 5). Sliding-window constraint.

SOFT

Prefer-off days

Penalty: 100 × role weight (PT=2.0, Manager=0.3). PT preferences are naturally protected more strongly.

SOFT

Shift time preference

Penalty: 50 × role weight per mismatch. Mismatched cells are highlighted ⚠️ in the dashboard.

SOFT

Senior coverage

Penalty: 150 per shift without a senior employee. Preferred but not mandatory.

Phase 2 — Fairness & Compliance

HARD

Evening→Morning interval

x[e,d,evening] + x[e,d+1,morning] ≤ 1. Prevents 8h turnarounds.

HARD

No consecutive evening shifts

No employee works 2+ evenings in a row. Prevents late-night fatigue accumulation.

HARD

Weekly hours cap

FT ≤ 40h (5 shifts), PT ≤ 20h (2–4 shifts per employee setting).

SOFT

Evening shift equalization

Penalty: 80 × (max−min evening count). Prevents evening shifts concentrating on the same people.

SOFT

Weekend shift equalization

Penalty: 80 × (max−min weekend count). Sat/Sun distributed as evenly as possible.

SOFT

Safety score threshold

Penalty: 800/skill/shift below threshold. Normal ≥40 / Holiday ≥50 / Sale ≥55.

v2 Feature

Training Plan Integration

When a Training Plan exists (from Training Planner), the optimizer automatically embeds OJT pairs as additional constraints — no manual configuration needed.

OJT — Trainee

Trainee = Off-strength (🟡)

During the training period, the Trainee's skill level is set to "none" for skill coverage calculations. They are present on the shift but do not contribute to Advanced/Basic coverage counts. Demand must still be met by the remaining staff.

OJT — Trainer

Trainer = Skill downgraded (⭐)

For Advanced training, the Trainer's skill is temporarily treated as "basic" (they are occupied teaching). For Basic training Phase 2, the Trainer skill is unchanged. This accurately models the reduced operational contribution during active OJT.

OJT — Pairing

Same-shift soft constraint (weight 5000)

The optimizer tries to assign Trainer and Trainee to the same shift each working day. Weight 5000 makes this effectively a near-hard constraint while still allowing separation if doing so would cause an infeasibility.

SOFT

Best-effort mode available

If "Best Effort" is toggled ON, minimum headcount constraints are relaxed to allow understaffed solutions. Useful for weeks when OJT pairs are unavoidable but demand drops. The dashboard shows clearly where demand is unmet (⚠️).

v2 Feature

Shift Stress Analysis

After optimization, a composite stress score is computed per employee and stored in session_state — feeding into the HR Manager and Ops dashboards.

Component 1
Shortage Stress (×0.5)
Headcount shortage ratio × role burden factor × day type factor × rare skill factor. MG/ASM=1.8×, SL=1.5×, Staff=1.2×, PT=0.8×. Sale days=1.4×, holidays=1.3×.
Component 2
Preference Stress (×0.2)
Shift preference mismatch rate. For "Any" preference employees, shift band variance is used as a proxy (max 0.3). Keeps the score meaningful for all staff types.
Component 3
Overtime Risk (×0.3)
MG / ASM / SL only. Shortage ratio × 0.6 + tenure factor × 0.2 + engagement factor × 0.2. Short tenure = 0.9 (can't refuse), long tenure = 0.3 (can push back). Also estimates overtime hours.
total_stress = shortage_stress × 0.5 + preference_stress × 0.2 + overtime_risk × 0.3

🟢 Low : score < 0.3
🟡 Mid : 0.3 ≤ score < 0.6
🔴 High : score ≥ 0.6

This stress data feeds the HR Manager dashboard (weekly stress summary + Early Warning) and the Retention Optimizer (overtime features for churn prediction).

Safety Score

Measuring skill quality per shift

Safety = (Advanced × 2 + Basic × 1) ÷ (Target headcount × 2) × 100

Using target headcount (not actual headcount) as denominator is intentional — a shift with 2 Advanced staff still scores lower than a fully staffed shift, because 2 people cannot safely cover a 6-person slot. Understaffing risk is built into the score.

Results

Total shifts, Penalty Score (0 = perfect), Total Hours, Weekly Wage, Evening gap, Weekend gap.

Staff per shift per day. Cells below demand flagged ⚠️.

⑤ Safety Score Heatmap

Skill × (day × shift) colored by safety score, with 🟢🟡🔴 relative to day-type threshold.

⑥ 😰 Shift Stress Analysis (v2 new)

Stacked bar chart: shortage (red) + preference mismatch (orange) + overtime risk (purple) per employee, sorted highest to lowest. OT risk warning for MG/ASM/SL ≥ 0.5.

When training plan active: Trainer/Trainee pair table + day-by-day shift assignment for each pair. Shows whether same-shift constraint was honored each day.

⑧ Substitute Suggestion

Select absent employee and shift — instant ranked replacement list filtered by hard constraints, ranked by skill match → hours slack → wage.
