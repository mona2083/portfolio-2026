OJT Phase Design

How training phases work

Each training type has distinct phases that affect how the Trainer and Trainee count in shift coverage calculations. This is what makes the SO integration non-trivial.

Training Type
Trainer effective skill
Trainee effective skill
Basic — Phase 1 (2 weeks)
none (full attention on teaching fundamentals)
none (learning, not yet operational)
Basic — Phase 2 (remaining weeks)
basic (supervising while operational)
none (still under supervision)
Advanced (all weeks)
basic (coaching while partially operational)
none (developing advanced skill)
Impact on shift coverage per active OJT pair:
Basic Phase 1: Trainer loses Basic skill (-1 effective) | Trainee = none
Basic Phase 2: Trainer stays Basic | Trainee = none
Advanced: Trainer drops Advanced → Basic (-1 tier) | Trainee = none

→ Each pair effectively reduces Advanced coverage count by 0.5–2 people

Capacity Analysis

Solving the trainer-shortage dilemma

The classic chicken-and-egg problem: "we need more Advanced staff, but we don't have enough Advanced staff to train them." A 4-tier analysis resolves this by separating "safe to train" from "train + hire simultaneously."

🔴
Critical

Coverage < 70%

Hiring only. Starting any training now would push coverage into dangerous territory. The team is already too thin.

🟠
Deficit

Coverage 70–90%

1 pair training + hire simultaneously. The chicken-and-egg resolution: start 1 pair to build future capacity while hiring to stabilize the present.

🟡
Marginal

Coverage 90–93%

1 pair maximum. Just above the safety threshold. Training is feasible but the buffer is thin — a single absence could push coverage below target.

🟢
OK

Coverage ≥ 93%

N pairs simultaneous. Calculated as: floor(buffer / impact_per_pair). The team has enough slack to absorb the training overhead.

buffer = current_hit_rate − (target_coverage + safety_margin) [default: 90% + 3%]
impact_per_pair = attend_rate × (hit_rate / adv_avg) × advanced_reduction_per_pair
max_pairs = floor(buffer / impact_per_pair) [min 0, overridden by tier logic]

Auto Plan Generation

From WP recommendation to OJT roadmap

One click generates the full training plan from Workforce Planner action items — no manual input required.

  1. Read WP action plan

    The planner reads wp_export_raw from session_state. "Train" actions identify which skills need development. "Hire" actions create placeholder entries for new hires (Trainer=TBD).

  2. Select Trainee candidates

    Candidates are filtered by: (1) current skill level matches the from-level for the training type, (2) not MG/ASM role (to preserve management capacity). Priority: FT/PT first, then SL.

  3. Select Trainer candidates

    For Basic training: FT Advanced holders preferred (not MG/ASM). For Advanced training: SL with Advanced preferred, ASM as backup. Trainer must not be in another active training pair.

  4. Schedule sequentially

    Plans are generated in serial — the next plan starts at ready_week + 1 of the previous plan. This prevents over-committing the same Trainer to multiple simultaneous sessions.

  5. Export to Shift Optimizer

    The complete training plan is serialized to JSON and passed to the SO via session_state. The SO reads it and applies OJT skill adjustments and pair co-scheduling constraints automatically.

5-Tab Interface

What each tab provides

① Training Plan Creation

Manual or auto-generated OJT plans. Each entry: Trainee, Trainer, skill, start week, estimated completion. Gantt chart visualization. Skill distribution table shows current state at a glance.

② Hiring Plan Creation

New hire entries with role, join week, skills to develop, and assigned OJT Trainer. The hire is counted as "none" in shift coverage until OJT completion week.

③ Progress & Edit

Live tracking of each training item: remaining hours, completion percentage, current week vs. ready week. Mark items complete, adjust start weeks, or reassign Trainers.

④ Summary & Export

Readiness calendar showing when each plan completes relative to seasonal peak deadlines (🟢 on time / 🟡 warning / 🔴 too late). JSON export button for Shift Optimizer integration.

⑤ Training Capacity Analysis

Heatmap: skill × pairs → coverage %. MC precision analysis: run precise simulation for specific skill. Recommended pairs table: Trainee + Trainer + recommended shift band + ✅/⚠️ status per coverage tier.

Why It Matters

What this demonstrates

🐣
Solving a Real Dilemma

The trainer-shortage problem is a real operational challenge in retail. The 4-tier capacity model doesn't avoid it — it names it, quantifies it, and provides a concrete resolution path.

🔗
Bidirectional Integration

TP consumes WP output (action plan) and produces SO input (training plan JSON). It's both a consumer and a producer in the pipeline — the architectural link between planning and execution.

📐
OR Applied to People

The capacity analysis uses linear approximation of MC simulation results — applying Operations Research methods to a staffing problem that most managers handle informally and incorrectly.

📅
Time-Aware Scheduling

Training plans are scheduled relative to seasonal peak deadlines. The readiness calendar answers the real question: "Will we have enough Deli Advanced staff before the Holiday rush?"
